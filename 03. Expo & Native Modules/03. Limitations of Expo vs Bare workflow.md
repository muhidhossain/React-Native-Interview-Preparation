Expo Managed workflow is highly convenient, but it comes with trade-offs. Key limitations include:

---

## 1. Limited Native Module Flexibility

You can only use native modules supported by Expo.
If a library requires custom native code (Objective-C/Swift/Java/Kotlin), you may need to eject.

**Example:**

- BLE (Bluetooth) deep integrations
- Advanced background services
- Custom camera modules
- Proprietary SDKs that need native linking

---

## 2. Some Features Depend on Expo Updates

You rely on Expo SDK releases. If Expo doesn't support a new iOS/Android API yet, you may have to wait.

**Example:**
A new Android permission system is released â†’ You can't integrate until Expo adopts it.

---

## 3. App Size Can Be Larger

Expo packages many libraries by default, even if your app doesn't need them â€” resulting in bigger APK/IPA.

---

## 4. Build Customization is Limited

You cannot fully customize:

- Gradle configs
- Info.plist / AndroidManifest beyond allowed scope
- Native pipeline and compression settings

**If business requires deep system configuration â†’ Bare workflow works better.**

---

## 5. Third-Party Library Incompatibility

Any package that requires custom native installation may not work directly.
